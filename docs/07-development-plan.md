# 开发阶段计划

> 原则：**先做完整闭环，再做展示亮点**。毕业设计不是拼功能数量，而是要做出一套从模型配置到对话、再到知识增强和数据管理都能跑通的桌面系统。

## 交付分层

### 最低可交付版本

满足答辩和论文撰写的最小范围：

- 多 Provider 基础接入
- 流式对话与消息持久化
- 会话管理 + 消息编辑/重发/重新生成
- 助手 / System Prompt / 参数设置
- 基础 RAG
- 1 到 2 个工具调用
- 本地导入导出

### 标准毕业设计版本

在最低可交付版本基础上，增加 2 到 3 个亮点：

- 多模型对比
- 自定义渠道
- TTS 或翻译
- URL 抓取入库

---

## Phase 1：基础框架与数据层 (3-4 天)

- [ ] 初始化 Tauri v2 + React 项目骨架
- [ ] 建立目录结构、配置管理、统一错误类型
- [ ] SQLite 初始化与迁移体系
- [ ] 对话、消息、助手、配置等核心表
- [ ] API Key 安全存储（keyring）

## Phase 2：聊天核心与 Provider 接入 (5-7 天)

- [ ] 定义统一 Provider trait 和消息类型
- [ ] OpenAI Adapter（基准实现）
- [ ] Claude 或 Gemini Adapter（二选一优先落地，再补另一家）
- [ ] Ollama / OpenAI-compatible Adapter
- [ ] 流式输出、取消生成、错误回传
- [ ] ChatService：上下文组装、发送、结果落库
- [ ] 消息编辑 / 重发 / 重新生成 / 多版本切换

## Phase 3：助手、提示词、参数系统 (3-4 天)

- [ ] PromptService：System Prompt 管理与变量渲染
- [ ] AssistantService：预设助手 + 自定义助手
- [ ] ParamService：标准参数、自定义参数、参数预设
- [ ] NamingService：自动生成会话标题
- [ ] 设置页与模型配置页联调

## Phase 4：RAG、工具调用与数据能力 (5-7 天)

- [ ] 文档上传、解析、分块、向量化、检索
- [ ] 对话中注入 RAG 上下文并展示引用来源
- [ ] Function Calling 主流程
- [ ] 内置工具：网页搜索或计算器
- [ ] URL 抓取正文，作为知识源或临时上下文
- [ ] 对话导出、本地数据导入/导出

## Phase 5：前端主界面与演示增强 (5-7 天)

- [ ] 会话列表、聊天窗口、输入区、设置页
- [ ] Markdown / 代码高亮 / 基础富文本渲染
- [ ] 助手面板、参数预设面板、知识库面板
- [ ] ToolCallCard、CitationTag 等核心交互组件
- [ ] 从以下增强项中选择 2 到 3 个完成：
- [ ] 多模型对比
- [ ] 自定义渠道
- [ ] TTS
- [ ] 翻译面板

## Phase 6：收尾、测试与答辩材料 (3-4 天)

- [ ] 统一异常提示与空状态
- [ ] 核心流程联调与回归测试
- [ ] 性能和资源占用基本验证
- [ ] 安装包打包
- [ ] 补齐架构图、流程图、数据库图、答辩演示脚本

---

## 不进入主线的功能

以下功能保留为后续扩展，不建议在毕业设计主周期内硬做：

- 对话分支（Fork）
- Artifacts 沙箱
- WebDAV 同步
- STT 语音输入
- PDF 导出
- Token 统计面板
- 复杂 Prompt 版本控制
- 过多内置工具生态

---

## 总工期建议

- 最低可交付版本：约 20-25 天
- 标准毕业设计版本：约 26-32 天

如果中途时间吃紧，优先保留 Phase 1-4，再从 Phase 5 的增强项里只留一个最能展示亮点的功能。
