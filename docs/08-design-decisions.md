# 关键设计决策 & 非功能需求

## 范围控制原则

毕业设计版本遵循以下原则：

1. 先保证“可用系统”而不是“功能百科全书”。
2. 每个大类功能至少有一个代表性实现，但不追求一次做满所有变体。
3. 优先做能写进论文、能现场演示、能体现架构能力的功能。
4. 高复杂度且对主线价值有限的功能，明确延期，而不是半成品上线。

## 关键设计决策

### 为什么用 Rust + Tauri 而非 Electron？

| 指标 | Tauri v2 | Electron |
|------|---------|---------|
| 空闲内存 | ~10 MB | ~150 MB |
| 安装包大小 | ~5 MB | ~80 MB |
| CPU 空转 | 接近零 | 明显 |
| 安全模型 | 细粒度权限声明制 | 全权限 |

Rust 适合处理并发网络请求、流式解析和本地数据处理，也便于与系统能力集成（如密钥管理、TTS）。

### 为什么不用统一的 OpenAI-compatible 层？

Claude 的 Messages API 和 Gemini 的 API 在以下方面差异较大：
- **消息格式**：system 字段处理方式不同
- **工具调用格式**：schema 结构完全不同
- **流式协议**：事件字段名不同
- **多模态**：图片传入格式不同

采用**统一 trait + 独立 Adapter** 保留灵活性，同时通过 OpenAI-compatible 基类减少重复实现。

### 自定义渠道为什么分阶段实现？

毕业设计里自定义渠道是亮点，但不适合一开始就做成“万能脚本平台”。

推荐分两步：

- 第一步：先支持 OpenAI-compatible 渠道配置
- 第二步：有余力再补模板 + JSONPath 映射

这样既能体现扩展性，也不会过早陷入复杂配置编辑器开发。

### 参数预设方案（差异化）

竞品多数只提供基础参数滑块。  
OmniChat 保留“精确 / 平衡 / 创意”等命名参数预设，并支持少量自定义参数透传，这个复杂度适中，也能体现产品设计思考。

### 消息版本系统为何只做线性版本？

对话分支（从任意节点分叉）虽然功能丰富，但会明显抬高实现成本：
- 数据模型复杂（树形消息 vs 线性消息）
- 历史组装逻辑复杂（需树遍历）
- 前端渲染复杂（分支导航 UI）

**决策**：毕业设计版本只做**线性版本**，即编辑历史消息、重新生成回复、切换同位置多个回答版本。这样既保留可演示性，也不把系统复杂度抬得过高。

### 为什么保留 RAG，但弱化记忆系统？

RAG 更容易形成稳定、清晰、可演示的闭环：

- 输入文档
- 建立索引
- 检索相关片段
- 在回答中展示引用来源

相比之下，长期记忆系统在毕业设计里更像增强功能，不适合作为主线依赖。

### API Key 安全存储

使用操作系统原生密钥管理：
- Windows → Credential Manager
- macOS → Keychain
- Linux → libsecret (GNOME Keyring / KWallet)

数据库中只存 key 的标识名，不存明文。内存中使用后即清除。

### RAG 为什么优先本地方案？

- 桌面应用的核心价值是**数据本地化**
- ONNX Runtime 在 CPU 上足以处理文档嵌入（每秒数百个 chunk）
- 作为可选项提供远程 Embedding API

### 工具调用为什么只保留少量内置工具？

工具调用本身已经足够体现系统的 Agent 能力。毕业设计版本保留 1 到 2 个可稳定演示的工具即可，例如：

- 网页搜索
- 计算器

MCP 可以保留接口和基础接入能力，但不建议在首版做成庞大的工具生态管理器。

## 后续扩展功能（本轮不做）

| 功能 | 裁剪理由 |
|------|---------|
| 对话分支 (Fork) | 树形数据结构和 UI 成本偏高 |
| Artifacts 沙箱 | 需要额外 WebView 安全设计 |
| AI 绘图 | 不影响聊天主流程，可后置 |
| WebDAV 备份 | 本地导入导出已满足毕业设计主需求 |
| STT 语音输入 | 平台差异明显，联调成本高 |
| PDF 导出 | 导出 Markdown/JSON 已可满足展示 |
| Token 用量统计 | 有价值，但不是主链路 |
| Prompt 版本历史 | 对首版收益有限 |
| 复杂工具市场 | 维护成本高，答辩收益一般 |

## 非功能性需求

| 指标 | 目标 |
|------|------|
| 冷启动时间 | 尽量控制在 2-3 秒内 |
| 空闲内存占用 | 明显低于同类 Electron 客户端 |
| 活跃对话内存 | 常规对话场景稳定运行，无明显卡顿 |
| 流式首 token 延迟 | 主要由网络决定，客户端不引入明显额外阻塞 |
| 大规模数据 | 支持长期使用场景下的消息存储与检索 |
| 安全 | API Key 加密，HTTPS 通信 |
| 跨平台 | Windows 10+ / macOS 12+ / Linux (major distros) |
